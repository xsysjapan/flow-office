# 11. バックオフィス処理ユースケース

## UC-B001: バックオフィスタスクを自動作成する

1. 申請が最終承認される
2. 申請種別設定を確認する
3. バックオフィスタスクが必要なら作成する
4. 処理部署、初期ステータス、期限を設定する
5. 担当者候補へ通知する

## UC-B002: 担当者を割り当てる

1. バックオフィス担当者が未担当タスク一覧を開く
2. 対象タスクを選択する
3. 担当者を割り当てる
4. ステータスを確認中にする
5. 担当者へ通知する

## UC-B003: 処理ステータスを更新する

1. 担当者が自分のタスクを開く
2. 処理状況を確認する
3. ステータスを更新する
4. コメントを残す
5. 履歴を記録する

### ステータス例

- 未着手
- 確認中
- 不備あり
- 処理中
- 発注済
- 支払予定
- 発送済
- 完了
- 取消

## UC-B004: 経費精算を処理する

1. 経理担当者が承認済み経費を確認する
2. 領収書、金額、用途、勘定科目を確認する
3. 不備があれば差戻しする
4. 問題なければ支払予定にする
5. 会計CSVまたは振込CSVを出力する
6. 支払後に完了にする

## UC-B005: 名刺申請を処理する

1. 総務担当者が承認済み名刺申請を確認する
2. 氏名、部署、役職、メール、電話番号を確認する
3. 発注データを作成する
4. 発注済みにする
5. 納品後、発送済みにする
6. 完了通知を送る

## UC-B006: 備品・グッズ申請を処理する

1. 総務担当者が承認済み申請を確認する
2. 在庫を確認する
3. 在庫があれば引当する
4. なければ発注済みにする
5. 発送または手渡し後に完了にする

## 実装上のポイント

- バックオフィス処理は承認とは別ステータス系列として管理する
  (`workflow_requests.status` と `backoffice_tasks.status` は独立)。
- タスク種別 (`task_type`) ごとに処理フロー・ステータス遷移が異なるため、
  ステータス遷移のバリデーションはタスク種別単位で定義する。
- `source_type` / `source_id` で `workflow_requests` など発生源に紐づく (ポリモーフィック)。
