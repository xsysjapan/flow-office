# 7. 勤怠管理ユースケース

## UC-A001: 出勤する

1. 社員が今日の勤怠画面を開く
2. 今日の勤務予定を表示する
3. 「出勤」ボタンを押す
4. 出勤日時を記録する
5. 日次勤怠ステータスを勤務中にする
6. `attendance.clocked_in` イベントを記録する

「今日」の判定は、サーバーやUTCの日付ではなく **その社員自身の `users.timezone` における
現在の日付** を基準に行う(休憩開始・休憩終了・退勤・打刻ログの業務日判定も同様)。理由は
docs/03-architecture.md 3.4 を参照。

## UC-A002: 休憩開始する

1. 社員が「休憩開始」を押す
2. 休憩開始日時を記録する
3. ステータスを休憩中にする
4. `attendance.break_started` イベントを記録する

## UC-A003: 休憩終了する

1. 社員が「休憩終了」を押す
2. 休憩終了日時を記録する
3. ステータスを勤務中に戻す
4. `attendance.break_ended` イベントを記録する

## UC-A004: 退勤する

1. 社員が「退勤」を押す
2. 退勤日時を記録する
3. 休憩時間、実働時間、深夜時間、休日労働時間、残業時間を再計算する
4. ステータスを退勤済みにする
5. `attendance.clocked_out` イベントを記録する

## UC-A005: 日次勤怠を編集する

1. 社員が日次勤怠画面を開く
2. 出勤、退勤、休憩、作業内容、備考を編集する
3. 修正理由を入力する
4. 保存する
5. 日次計算を再実行する
6. `attendance.day_edited` イベントを記録する

日次編集は締め前のみ可能とする。月次締め後は通常編集ではなく修正申請を使う。また、月次が
承認済み(UC-A009)以降は、締め(UC-A011)がまだ行われていなくても通常の日次編集はできない
(UC-A015参照)。承認後はバックオフィス確認対象になるため、承認時点の内容から日次側の記録を
変更させないためである。

出勤・退勤・休憩の時刻は、社員本人の既定タイムゾーン(`users.timezone`)ではなく、その
勤務日に指定したオフセットのまま保存・表示する。海外出張などで現地時刻が既定タイムゾーンと
異なる場合、編集時に現地のオフセット付きISO8601を入力することで、その勤務日だけ現地時刻で
記録できる(例: ニューヨーク出張中に現地22:00〜翌05:00勤務した場合、`-05:00`のオフセットで
入力する。深夜時間(22:00〜05:00)の判定もこの現地時刻を基準に行われる)。1回の編集で送る
出勤・退勤・休憩の時刻は全て同じオフセットである必要があり、異なるオフセットが混在する場合は
エラーになる(docs/03-architecture.md 3.4)。

## UC-A006: 週次勤怠を編集する

1. 社員が週次勤怠画面を開く
2. 7日分の日次勤怠を一覧表示する
3. 未入力、打刻漏れ、休憩不足、長時間労働を警告表示する
4. 複数日をまとめて編集する
5. 保存時は日次単位の編集イベントに分解して記録する
6. 週次合計を再計算する

週次勤怠は独立データではなく、日次勤怠の編集ビューとする。

## UC-A007: 月次勤怠を確認する

1. 社員が対象月を選択する
2. 月内の日次勤怠一覧を表示する
3. 実働、所定内、法定内残業、法定時間外、深夜、休日労働、有給を集計する
4. 未入力や警告を表示する
5. 問題がある場合は該当日の日次編集へ遷移する

## UC-A008: 月次勤怠を提出する

1. 社員が月次勤怠を確認する
2. エラーがないことを確認する
3. 承認者を任意の社員から選択する
4. 月次提出する
5. 提出時点の集計スナップショットを保存する
6. 月次ステータスを提出済みにする
7. 承認者へ Teams 通知を送る
8. `attendance.month_submitted` イベントを記録する

## UC-A009: 承認者が月次勤怠を承認する

1. 承認者が承認待ち一覧を開く
2. 対象社員の月次勤怠を確認する
3. 日別明細、月次集計、警告を確認する(シフト制勤務者の場合、法定休日要件の警告を含む。
   UC-C005参照)
4. 承認する
5. 月次ステータスを承認済みにする
6. バックオフィス確認対象にする
7. `attendance.month_approved` イベントを記録する

## UC-A010: 承認者が月次勤怠を差戻しする

1. 承認者が勤怠内容を確認する
2. 不備日とコメントを入力する
3. 差戻しする
4. 月次ステータスを差戻しにする
5. 社員へ Teams 通知を送る
6. 社員は修正後に再提出する

## UC-A011: 管理部が月次勤怠を締める

1. 管理部が対象月を選択する
2. 未提出、未承認、警告ありの社員を確認する
3. 問題がなければ締め処理を実行する
4. 対象月の日次勤怠をロックする
5. 給与計算用CSVを出力可能にする
6. `attendance.month_closed` イベントを記録する

## UC-A012: 打刻ログを記録する

1. デバイス(ICカード端末、モバイル端末など)またはWeb画面が打刻イベント
   (出勤・休憩開始・休憩終了・退勤のいずれか)と、それが属する業務日(`work_date`)、
   実際に発生した日時(`punched_at`)、打刻元(`source`)を送信する
2. 矛盾の有無を検証せずに、必ず打刻ログとして記録する
3. `attendance_punch.recorded` イベントを記録する
4. 同一社員・同一業務日の打刻ログ全体を集めて、矛盾なく1日分の勤務
   (出勤1件・退勤1件・0件以上の休憩が正しい順序で収まっている)として
   組み立てられるか判定する
5. 組み立てられる場合のみ、対象の日次勤怠 (`attendance_days` / `attendance_breaks`) に反映し、
   日次計算を再実行する。ただし対象日が既に画面の出退勤操作または日次編集で確定している場合は
   上書きしない
6. 組み立てられない場合(打刻漏れ・重複・順序不整合など)は、打刻ログの記録のみ行い、
   日次勤怠には反映しない。矛盾の解消は社員自身または管理者がUC-A005の日次編集で行う

打刻ログ (`attendance_punches`) は勤怠の正ではなく **参考情報** である。矛盾があっても
記録自体は必ず成功させる(打刻元デバイスや通信経路の都合で、出勤のみ・退勤のみが単独で
届く、同じイベントが重複して届く、順序が入れ替わって届くことを前提とする)。日次勤怠こそが
最終的な整合性を保証する正データであり、打刻ログはそれを補助する参考記録に過ぎない。

打刻は必ず特定の業務日 (`work_date`) に明示的に所属させる。例えば21:00に出勤し翌6:00に
退勤する夜勤の場合、`punched_at` の日付(カレンダー上の日付)ではなく、その勤務がどちらの
勤務日に属するかを打刻元が判断して `work_date` に指定する(この例では出勤時刻の日付、
すなわち夜勤当日に両方の打刻を所属させる)。

`punched_at` はAPIの入力・出力ともオフセット付きISO8601で送受信する(オフセットなしの
日時は入力として拒否する)。社員本人の既定タイムゾーン(`users.timezone`)には変換せず、
打刻元デバイスが送った通りの現地時刻とオフセットをそのまま記録する(海外出張中に現地の
ICカード端末で打刻した場合、その現地時刻がそのまま保持される)。同一業務日の打刻は全て
同じオフセットである必要があり、異なるオフセットが混在する場合は矛盾ありとして扱う
(docs/03-architecture.md 3.4)。

## UC-A013: 打刻ログを訂正する

1. 社員本人または管理者が、対象の打刻ログ(誤打刻・打刻漏れの補完など)を選ぶ
2. 正しい打刻種別(出勤・休憩開始・休憩終了・退勤)と日時を入力する
3. 訂正理由を入力する
4. 保存する
5. 元の打刻ログは書き換えず「訂正済み」として残し、訂正後の値を新しい打刻ログとして
   追記する(打刻ログは追記のみで、既存の行を直接書き換えない)
6. `attendance_punch.corrected` イベントを記録する
7. 訂正後の打刻ログ一覧をもとに矛盾なく1日分の勤務として組み立てられるか再判定し
   (UC-A012 手順4)、組み立てられる場合のみ対象の日次勤怠に反映し直す。ただし対象日が
   既に画面の出退勤操作・日次編集で確定している場合、または締め後にロックされている場合は
   上書きしない

訂正前後どちらの打刻ログも、誰が・いつ・どのような理由で訂正したかとともに一覧で参照できる
(UC-A012の打刻ログ一覧に、状態(有効・訂正済み・削除済み)として表示される)。

対象日が属する月次が承認済み以降(承認済み・締め済み)の場合は、打刻ログの訂正自体ができない
(UC-A015と同じ`AttendanceEditGuard`の制約に従う)。打刻ログの状態(有効/訂正済み/削除済み)が
変わることは、承認済みの記録に対する監査証跡の書き換えに当たるため、日次勤怠の反映有無に
かかわらず一律に禁止する。修正が必要な場合は修正申請ワークフロー([10章](./10-usecases-workflow.md))
を使う。

## UC-A014: 打刻ログを削除する

1. 社員本人または管理者が、対象の打刻ログ(重複打刻など明らかな誤りを想定)を選ぶ
2. 削除理由を入力する
3. 削除する
4. 打刻ログの行は物理削除せず「削除済み」として残す(訂正と同様、削除も操作の履歴として
   理由・実行者・日時付きで参照できるようにする)
5. `attendance_punch.deleted` イベントを記録する
6. 削除後に残った打刻ログをもとに矛盾なく1日分の勤務として組み立てられるか再判定し、
   組み立てられる場合のみ対象の日次勤怠に反映し直す(UC-A013 手順7と同じ制約に従う)

UC-A013と同様、対象日が属する月次が承認済み以降の場合は打刻ログの削除自体ができない。

## UC-A015: 日次勤怠を削除する

1. 社員本人または管理者が対象の日次勤怠(勤務予定を誤って実績化した、二重に打刻・入力
   された、などの理由を想定)を選ぶ
2. 削除理由を入力する
3. 削除する
4. `attendance.day_deleted` イベントを記録する
5. 対象の `attendance_days` 行(および従属する `attendance_breaks` / 日次集計)を削除する

日次勤怠は、対象月が未提出・提出済み・差戻しのいずれかの間(=承認前)のみ削除できる。
月次が承認済み以降(承認済み・締め済み)は、締め(`locked_at`)の有無によらず変更・削除でき
ない(UC-A005参照。修正が必要な場合は修正申請ワークフロー([10章](./10-usecases-workflow.md))
を使う)。また、対象日に有給消化(`paid_leave_usages`)が既に反映されている場合は、有給残数の
整合性が崩れるため削除できない。`paid_leave_usages` は有給申請が承認された時点で作成され
(UC-P004)、承認済みの有給申請を取り消す機能は現状無いため、この場合は削除不可が最終的な
扱いになる。

削除は打刻ログの訂正・削除と異なり、行を保持しない(`stored_events` に記録される
`attendance.day_deleted` イベント自体が、誰が・いつ・どのような理由で削除したかの記録として
残る)。削除後、同じ勤務日に対して改めて打刻・日次編集・有給承認のいずれかの経路で
`attendance_days` を作成できる。

## UC-A016: 出勤日を新規作成する

1. 社員本人または管理者が、対象社員・対象の勤務日を指定する
2. 出勤・退勤・休憩(任意。いずれも未入力でよい)、作業内容、備考を入力する
3. 作成理由を入力する
4. 作成する
5. 日次計算を実行する
6. `attendance.day_created` イベントを記録する

打刻(`attendance_punches`)と出勤日(`attendance_days`)は、勤務日(`work_date`)が同じという
だけの緩い関係しかない。打刻の有無にかかわらず、出勤日はその月が編集不可(承認済み・締め済み。
UC-A005/UC-A015と同じ`AttendanceEditGuard`の制約)になるまでは、いつでも作成・削除できる。
「出勤操作」(UC-A001、ClockIn)は当日しか使えないため、過去日・未来日への作成や、
打刻を一切伴わない出勤日の作成(裁量労働制でみなし時間のみを計上する場合など。
docs/07-usecases-attendance.md「裁量労働制・管理監督者」参照)にはこちらを使う。

対象の勤務日に既に出勤日が存在する場合は作成できない(日次編集(UC-A005)を使う)。

## 週40時間判定(日8時間判定との二重計上排除、参考情報)

1日8時間超の法定時間外(`attendance_daily_calculations.statutory_overtime_minutes`)とは別に、
1週40時間(労基法32条)を超えた分を `WeeklyOvertimeCalculator` が月次確認画面の表示のたびに
日次実績から都度計算する参考情報として提供する(UC-C005の法定休日要件チェックと同じ考え方。
週次勤怠は日次勤怠の編集ビューであり、月のように独立した集計単位ではないため、Projectionとして
永続化せず、`attendance_months.snapshot_json`(月次提出スナップショット)にも合算しない)。

- 対象週の各日について、日8時間超で既に計上済みの時間を除いた「日8時間以内の実働」だけを
  合計し、40時間を超えた分のみを `weekly_statutory_overtime_minutes` とする(二重計上しない)。
- 法定休日労働はこの週40時間の判定に含めない(法定休日労働は別枠の休日割増で扱う)。
- 法定外休日(所定休日)の労働は、それだけを理由に休日割増は付けないが、日8時間・週40時間
  いずれの判定からも除外しない(`AttendanceCalculator`は法定休日のみを日次判定から除外し、
  法定外休日は通常の労働日と同様に扱う)。
- 月次確認画面(`AttendanceMonthResource`)には `weekly_overtime_reference` として、対象月に
  含まれる各週の実働・日次残業・週次残業を都度計算して表示する。あくまで参考情報であり、
  給与計算上の確定値としては扱わない。

## 裁量労働制・管理監督者

`attendance_daily_calculations` は実労働時間(`actual_work_minutes`、健康管理用)と、
給与計算上使用する労働時間(`payroll_work_minutes`)を分離して保持する。通常は両者が一致するが、
以下の勤務形態では異なる。

- **裁量労働制(`work_time_system=discretionary`)**: 対象日(所定の稼働日かつ法定休日でない日)
  は、実労働時間にかかわらず`work_styles.deemed_daily_minutes`(みなし時間)を
  `payroll_work_minutes`として採用する(`deemed_work_minutes`に記録)。みなし時間が8時間を
  超える設定の場合、超過分は毎稼働日の法定時間外として自動計上する。出退勤の打刻・実績入力は
  必須にしない(打刻が無い日も`deemed_work_minutes`の分だけ`payroll_work_minutes`が計上される)。
  ただし深夜労働・法定休日労働は、みなし時間ではなく実際に入力された時刻から計算する(実績が
  無ければ0として扱う)。法定休日の実労働は、みなし労働の対象日から除外され、実際の時刻から
  通常通り計算する。
- **管理監督者(`work_time_system=manager_supervisor`)**: 労働時間・休憩・休日の規定の適用が
  除外されるため、`non_statutory_overtime_minutes` / `statutory_overtime_minutes` /
  `legal_holiday_work_minutes` / `company_holiday_work_minutes` は常に0にする(残業・休日の
  割増計算対象にしない)。深夜割増(`late_night_minutes` / `legal_holiday_late_night_minutes`)
  および実労働時間の記録(健康管理)は引き続き適用・記録する。有給休暇の扱いも通常通り。

労使協定・本人同意の管理(協定の届出情報、本人同意の取得・撤回等)は本システムのスコープ外と
する(docs/08-usecases-calendar-shift.md UC-C002の注意点と同じ方針)。

## データの正 (再掲)

- 勤務予定: `employee_shift_assignments`
- 勤務実績: `attendance_days` / `attendance_breaks`
- 集計結果: `attendance_daily_calculations` (日次) / `attendance_months` (月次スナップショット)
- 打刻ログ (参考情報、正ではない): `attendance_punches`

月次勤怠 (`attendance_months`) は直接編集不可。締め後の `attendance_days` は
`locked_at` を設定してロックし、以降の修正は修正申請ワークフロー
([10章 汎用申請](./10-usecases-workflow.md)) を経由する。

`attendance_days.source` は `actual_start_at` / `actual_end_at` / 休憩がどの経路で最後に
確定したかを表す(`live`: 画面の出退勤操作、`manual`: 日次編集、`punch`: 打刻ログからの
自動反映)。`punch` 以外の経路で確定した日は、打刻ログが後から届いても上書きされない。
